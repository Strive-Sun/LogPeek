## Context
M1 是 LogPeek 的第一个可运行闭环,目标是最快验证核心假设:"压缩包丢进监控目录 → 自动识别 → 顶栏提示 → 免解压查看"。约束:仅 zip 与裸文本文件。技术栈已定为 Tauri 2.x(Rust 后端 + React/TS 前端),详见 `docs/technical-design.md`。

关于大文件与磁盘取舍:目标场景是高频下载,单个压缩包可达几百 MB(解压后可能 1GB+)。M1 采用**方案 A**——打开 Deflate 压缩条目时一趟顺序解压到内部临时缓存文件,换取打开后随机跳转/滚动的无感体验;代价是临时缓存占用等同解压后大小的磁盘(用完即清)。免磁盘放大的 **Deflate 重启点索引**留待 M2。

## Goals / Non-Goals
- Goals:
  - 打通监控 → 提示 → 免解压查看的端到端链路
  - 确立 `ArchiveReader` trait 抽象,为后续多格式扩展留接口
  - 确立前后端契约(invoke 命令 + emit 事件)
  - 行索引 + 窗口化加载,支持解压后 ≤ 2GB 的单文件流畅查看(方案 A:压缩条目落盘缓存)
- Non-Goals:
  - Deflate 重启点索引(免磁盘放大的随机访问)——M2
  - 采样索引优化(进一步降低索引密度)——M2
  - tar.gz / 7z / rar(M3)
  - 搜索、过滤、高亮(M4)
  - 注:多目录监控**属于 M1**(见 `directory-monitoring` 能力);主题在 M1 仅做 CSS 变量骨架(跟随系统 + 手动切换),配色打磨留待 M4

## Decisions

- Decision: 监控用 `notify` crate,支持同时监控多条目录;收到 create 事件后做**大小稳定检测**(轮询文件大小,连续 N 次不变视为到达完成),再判定格式。
  - Alternatives considered: 监听平台 close 事件——跨平台行为不一致,`notify` 封装不完整,故用大小稳定这种可移植方案。

- Decision: 监控目录列表持久化到本地配置文件(JSON,位于平台标准配置目录,如 `tauri::api::path::app_config_dir`)。启动时读取并恢复监控,增删即时写回。失效目录跳过不阻断启动。
  - Alternatives considered: 内存态不持久化——用户每次启动都要重设,体验差,排除。用系统数据库——M1 规模用不上,JSON 足够。

- Decision: 新日志计数采用"逐条递减 + 一键清零"。查看某个新包 → 计数减一(已看集合去重,重复查看不再减);"全部标记已读" → 计数归零。单纯点击提示只展开列表,不改计数。
  - Alternatives considered: 点击即全部清零——会让用户漏看未查看的包,已否决。

- Decision: 定义 `ArchiveReader` trait(`entries()` + `open_entry() -> Read 流`),M1 只实现 zip。
  - Alternatives considered: M1 直接写死 zip 逻辑不做抽象——但抽象成本很低且是核心扩展点,提前定义避免 M3 重构。

- Decision: 免解压 = zip 条目以 `Read` 流交给上层,不产生用户可见的解压产物。查看时后台流式扫描该条目建立行偏移索引,前端虚拟滚动只请求可视区行范围(`read_lines`),后端按偏移读取对应行返回。
  - Alternatives considered: 全量读入内存返回给前端——几百 MB 会卡死/OOM,排除。解压到用户可见目录——违背产品核心卖点,排除。

- Decision: **区分 zip 的两类条目做随机访问(M1 方案 A)**。Stored(未压缩)条目解压后偏移 = 归档内偏移,可对归档直接 seek,真正零落地。**Deflate(压缩,zip 默认)条目的解压流是顺序的,无法中途 seek**——M1 在打开时做一趟顺序解压,将解压结果写入**内部临时缓存文件**(非用户可见)并同时建立行偏移索引;此后随机跳转/滚动均等价于对普通未压缩文件 seek,无感。缓存随 session 关闭清理。索引在后台建立并通过 `index-progress` 事件反馈进度。
  - Trade-off: 临时缓存占用 = 解压后大小(几百 MB 压缩包可达 1GB+),换取打开后的随机访问无感。用完即清 + LRU + 退出兜底控制磁盘占用。
  - Alternatives considered: 假设所有 zip 条目都可直接 seek 解压后偏移——对 Deflate 流物理上不成立,是原设计的错误,已纠正。建索引前整体解压到内存缓冲——大文件 OOM,排除。**Deflate 重启点索引(每隔若干 MB 保存 inflate 器状态,免落盘随机访问)——免磁盘放大、更优,但需底层 inflate 接口、复杂度高,定为 M2**;M1 先用落盘缓存快速跑通闭环。

- Decision: 前后端契约用 Tauri `invoke`(前端→后端请求)+ `emit`(后端→前端推送)。新包到达是后端主动事件,故用 emit。
  - Alternatives considered: 前端轮询——延迟高、浪费,排除。

## Risks / Trade-offs
- 大小稳定检测有固有延迟(等待稳定窗口)→ 窗口设短(如 500ms×3),在延迟与误判间平衡。
- 行索引占内存:每行 8 字节偏移,1GB / 每行 100 字节 ≈ 1000 万行 ≈ 80MB 索引;2GB 上限约 160MB,可接受(采样索引进一步压缩留待 M2)。
- **临时缓存磁盘占用**:方案 A 下每个打开的压缩条目占等同解压后大小的磁盘(几百 MB~2GB),多会话叠加可达数 GB → LRU 上限(会话数)+ 关闭即清 + 进程退出兜底控制;若磁盘写入失败则回退为仅顺序读(不支持随机跳转)并提示。
- 单文件解压后 > 2GB → 拒绝建索引并提示"超大文件支持将在后续版本提供"(上限按实际读取字节数熔断,不信任声明值)。
- 建索引耗时(需扫全文件一遍)→ 后台线程建索引 + `index-progress` 进度事件 + 边建边读,不阻塞 UI。
- zip 内可能有非文本文件 → 按扩展名 + 采样判定 is_log,非文本条目仍列出但标记。

## Migration Plan
全新项目,无存量迁移。M1 已含行索引 + 窗口化加载(方案 A:压缩条目落盘缓存)。M2 在 `log-viewing` 能力上以 MODIFIED 方式引入 **Deflate 重启点索引**(免磁盘放大的随机访问)与采样索引优化,替换/补充落盘缓存路径。

## Open Questions
- 持久化配置文件的具体 schema(除目录列表外是否含窗口大小、上限等)——M1 先只存目录列表,后续按需扩展。
- "已看"状态是否也需持久化(重启后已看的包是否仍算已看)——M1 先做内存态,重启后计数从当前目录扫描重建;后续再评估。
