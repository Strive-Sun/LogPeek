## Context

`WatchState::start_watch` 当前在单个事件消费线程中对每条路径同步执行 `stable_detect`。该检测至少等待三个 500ms 稳定窗口，期间后续删除/重命名事件只能排队。检测成功后后端仅发送 `new-archive-detected`，前端收到该通知才调用 `list_watch_dirs`，而该命令会重新扫描全部监控目录并对未知扩展名读取内容样本。

需要同时解决正确性和性能：结构变化必须及时到达 UI，但批量解压或下载可能瞬间产生数百个底层事件，不能为每个事件全量扫描。

## Goals / Non-Goals

- Goals:
  - 当前已加载目录中直接子项的创建、删除、重命名和元数据修改最终与磁盘一致。
  - 文件夹作为可展开节点显示，展开时按需加载和监听，避免递归扫描整个监控根目录。
  - 结构同步不受日志后缀筛选、已读状态或稳定检测耗时影响。
  - 空闲时不轮询；事件突发时进行有界合并和局部更新。
  - Windows 与 macOS 的不同 rename/remove 事件形态产生一致结果。
- Non-Goals:
  - 不在启动时递归扫描或监听整棵目录树；未展开目录的内部变化在展开时通过快照获取。
  - 不实时展示仍在写入文件的内容；日志通知仍须等待稳定检测。
  - 不在本 change 中为超大目录引入完整目录树虚拟列表。

## Decisions

### 结构事件与日志到达分离

- watcher 收到原生事件后先走轻量结构同步路径，再把需要稳定检测的创建/修改路径放入独立后台队列。
- 结构路径处理当前目录的全部直接子项；目录节点不进入稳定检测，文件节点继续由日志到达路径负责 zip/文本识别、后缀规则和铃铛通知。
- 删除和重命名不等待文件稳定，避免旧节点滞留至少 1.5 秒或永久不消失。

### 事件标准化与批处理

- 后端定义可序列化的目录变化批次，至少包含监控目录、变化类型、旧/新路径和最新节点元数据。
- 同一监控目录在约 200ms 窗口内的原生事件合并为一个批次；同一路径的重复 modify 只保留最新状态。
- 可明确表达的 create/remove/rename/modify 直接作为增量变化发送；事件缺少路径配对、发生 watcher overflow/error 或平台语义不明确时，批次标记该目录需要 rescan。

### 前端局部状态更新

- API 层订阅独立的 `directory-changed` 事件。
- 纯函数 reducer 以绝对路径为稳定 id，对单个目录执行 upsert/remove/rename 或用 rescan snapshot 替换 children，并保持确定性排序。
- `App` 不再从新日志回调触发全目录 `refreshTree`；新日志列表与目录结构分别更新。
- 后缀筛选继续在渲染阶段执行，因此被隐藏文件仍存在于内存库存，切换“显示全部”时无需重新扫描。

### 懒加载递归目录

- 初始扫描只读取每个监控根目录的直接子项，文件夹节点初始不携带已加载 children。
- 用户展开文件夹时，前端调用后端读取其直接子项，并为该目录建立一个非递归 watcher；变化批次的 `watchDir` 指向该文件夹绝对路径。
- 用户折叠文件夹或其祖先被移除时，释放该目录及已展开后代的 watcher；再次展开以最新磁盘快照恢复状态。
- reducer 递归定位 `watchDir`，只替换受影响目录节点的 children，并保持其它分支引用稳定。

### 稳定检测并发与去重

- 稳定检测使用有界工作队列，按路径保留最新 generation；同一路径连续 modify 不创建无限任务。
- 文件删除或重命名使旧路径任务失效，完成较晚的旧检测结果不得重新插入节点或通知。
- 队列压力过高时优先保证结构同步，并对受影响目录安排一次兜底重扫，而不是无限增长内存。

### 快照与分类成本

- 初始扫描、展开扫描和兜底重扫枚举目标目录的直接文件与文件夹并读取基础 metadata，构造该层库存。
- zip/日志识别可对新增或变化文件异步完成；未知文件先作为普通文件节点存在，不能因分类失败从“显示全部”中消失。
- 不在每个结构事件上读取所有文件的 4KB 内容样本。

## Risks / Trade-offs

- Windows/macOS 对 rename 的事件组合不同 → 标准化层优先配对，无法可靠配对时局部 rescan。
- 批处理窗口带来约 200ms 的可见延迟 → 换取大量文件操作时稳定且低 IO，仍属于用户感知上的实时同步。
- 内存和 watcher 数量与“已加载节点/已展开目录”相关；极端十万级单目录仍需后续虚拟化或容量提示。
- 原生 watcher 可能丢事件 → error/overflow 和应用恢复焦点时可触发局部 reconciliation，保证最终一致。

## Migration Plan

无需持久化数据迁移。升级后首次启动仍执行现有目录初始扫描，随后由新结构事件接管同步；旧的监控目录和筛选配置继续使用。
