## Context

`App` 当前通过条件渲染 `fileSearchOpen && <FileSearchPanel />` 切换搜索页面。关闭搜索会销毁组件全部本地状态；再次打开时挂载副作用重新调用 `fileSearchStatus`、`fileSearchConfig` 并订阅状态，因此出现“搜索中”过渡。

Rust 初始化任务当前先调用 `FileSearchManager::new()`，该构造过程会创建目录、恢复 Tantivy 快照、打开查询索引、初始化 SQLite 并刷新计数；完成后才发布 `ReadyAppState`、恢复监控目录 watcher。即使搜索偏好关闭，构造过程仍会打开数据库和索引，推迟核心功能就绪。

## Goals / Non-Goals

- Goals:
  - 首次访问后在当前进程保留搜索页面状态。
  - 页面往返不触发由重新挂载导致的状态、配置或查询请求。
  - 隐藏页面不参与交互或可访问性导航。
  - UI、监控目录和日志会话优先完成初始化并可用。
  - 搜索关闭时不构造重型搜索管理器；搜索开启时在核心功能就绪后后台构造。
  - 搜索初始化不占用 UI 线程，也不阻塞非搜索命令。
- Non-Goals:
  - 跨应用重启保存搜索结果。
  - 停止后台索引状态订阅或文件索引更新。
  - 更改搜索算法、排序或结果上限。
  - 运行中动态启停搜索后端；开关仍在下次启动生效。

## Decisions

- Decision: `App` 增加“搜索页面是否曾打开”状态。第一次进入前不挂载搜索页面；第一次进入后始终保留组件，只通过可见状态切换。
- Decision: 隐藏容器使用平台原生 `hidden` 语义，确保布局、指针事件和可访问性树均不包含隐藏搜索页面。
- Decision: `FileSearchPanel` 接收当前激活状态。激活时聚焦搜索框，隐藏时恢复进入搜索前的焦点；初始化 API 和订阅副作用仍只在组件生命周期执行一次。
- Decision: 不使用 `localStorage` 或数据库保存结果集合。已有搜索索引负责跨重启数据持久化，界面查询状态只在进程内保活，避免显示过期结果。
- Decision: 将搜索偏好读取和写入提取为轻量存储，不依赖 `FileSearchManager`。启动时先读取一次启用偏好并冻结为本次进程状态。
- Decision: `ReadyAppState` 先发布监控、会话和搜索运行时占位状态，随后恢复监控目录 watcher。搜索运行时使用一次性异步就绪单元保存管理器，使非搜索命令无需等待搜索初始化。
- Decision: 只有冻结状态为启用时，才在 watcher 恢复后通过 `spawn_blocking` 构造 `FileSearchManager` 并恢复索引/provider；冻结状态为关闭时本次进程不构造管理器。
- Decision: 搜索状态、配置、控制和查询命令在本次搜索启用但管理器尚未完成构造时等待搜索就绪；读取/保存下一次启动偏好的命令始终使用轻量存储并立即可用。

## Risks / Trade-offs

- 隐藏页面仍占用最多 1000 条结果对应的前端内存 → 沿用现有结果上限，换取即时返回体验。
- 隐藏期间索引状态仍可能更新页面状态 → 保留订阅，使重新进入时直接显示最新状态，而不是重新加载。
- 搜索已启用时，用户首次进入搜索页可能遇到后台仍在初始化 → 仅搜索命令等待就绪，并保持可读加载状态；UI 和监控目录不受影响。
- 偏好存储与管理器配置分离可能发生写入竞争 → 两者共享同一原子配置写入实现，管理器初始化读取冻结后的已保存配置快照。
