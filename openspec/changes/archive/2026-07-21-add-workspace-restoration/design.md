# Design: 工作区恢复与源文件变化处理

## 持久化模型

前端使用带版本号的本地工作区快照，仅保存稳定信息：条目 key、可见顺序、折叠顺序和活动 key。会话 ID、索引缓存、加载状态和错误不持久化。读取快照时执行类型校验、去重和数量限制；损坏或未知版本数据回退为空工作区。

恢复出的选项卡统一为 `dormant`。启动时验证每个 key 对应的最外层磁盘文件是否存在，移除缺失项并提示；只自动打开活动选项卡，其余选项卡在首次激活时惰性打开。

## 源文件身份

归档条目链 `outer.zip::inner.tar::app.log` 的磁盘源始终是最外层 `outer.zip`，裸日志的磁盘源就是自身。后端提供只读文件修订查询，返回存在性、大小和修改时间；打开成功后前端保存当时修订，用于去重同一轮 watcher 事件。

## 运行时变化

- `upsert` 命中已打开选项卡的磁盘源时，将其视为潜在修改并查询最新修订；修订未变化不提示。
- 同一源文件只排队一个变化提示，避免多个归档条目或事件风暴重复弹窗。
- 确认重新加载时关闭旧会话并重新打开活动条目；同源其它条目释放旧会话并转为休眠。
- 忽略修改时记录最新修订作为已知状态，继续显示当前缓存快照；后续再次变化仍会提示。
- `remove` 命中磁盘源时标记同源选项卡为 `deleted`，提示一次但不关闭会话。若同源存在会话缓存，优先选择当前活动条目，否则选择该源下第一个可用缓存，提示提供“不保存/另存为”；没有缓存时只提供“知道了”。`deleted` 选项卡不持久化。
- 打开会话后保存适配层实际解析出的最外层磁盘源绝对路径，避免归档显示 key 与 watcher 绝对路径无法匹配。活动选项卡另以低频文件状态查询兜底，并在窗口重新获得焦点时立即查询，以覆盖操作系统 watcher 偶发漏报。
- 应用自身删除前登记抑制标记，消费对应 watcher 删除事件后清除，避免双重提示。

## 对话框队列

文件变化可能与删除确认或 updater 提示同时发生。工作区维护 FIFO 文件变化队列，每次只展示一个应用内对话框；修改对话框提供“保持当前内容”和“重新加载”，删除对话框在存在会话缓存时提供“不保存”和“另存为”，否则只提供“知道了”。

## 删除后另存为

后端按 session ID 导出已经写入缓存的原始字节，并返回导出字节数及索引是否完成。导出前获取缓存当前稳定长度，之后最多复制该长度，避免与仍在进行的追加索引相互污染。系统保存窗口必须选择新路径，不得默认写回已经删除的源路径。

归档条目导出的是解压后的条目内容，不是重新创建归档。索引尚未完成或已失败时允许导出当前缓存部分，但完成后必须明确提示这是不完整快照。

## 边界

- 不持久化日志正文缓存，因此应用关闭期间发生修改时，重启后直接读取磁盘当前版本。
- 不恢复精确滚动位置和手动编码；这两项保留为后续增强。
- 不恢复已删除源文件，也不覆盖原路径；仅允许将现有会话缓存另存为新文件。
