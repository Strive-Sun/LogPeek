# Windows 启动性能记录

## 测试环境

- 日期：2026-07-24
- 系统：Windows 10 专业版 10.0.19045
- CPU：Intel Core i7-10700K 3.80 GHz
- 内存：15.8 GB
- 构建：`cargo build --release --bin logcrate --features custom-protocol`
- 搜索配置：已开启，根目录为 `C:\`、`D:\`
- 测量点：进程入口到 React 主页面加载层移除后的首个绘制帧

## 优化前阶段基线

接入阶段计时但尚未加入可交互门控时运行 3 次：

| 样本 | 可交互耗时 |
| --- | ---: |
| 1 | 1469.8 ms |
| 2 | 1608.0 ms |
| 3 | 1739.6 ms |

P50 为 1608.0 ms，最大值为 1739.6 ms。后台 watcher 与搜索在约 371 ms 时已经开始调度，早于约 1.6 秒的可交互帧，存在与 WebView 冷启动争用 CPU、磁盘和数据库锁的风险。

## 优化后正式基准

通过 `npm run startup:bench -- --runs 10` 连续运行 10 次：

| 样本 | 可交互耗时 |
| --- | ---: |
| 1 | 1069.2 ms |
| 2 | 1359.2 ms |
| 3 | 1154.4 ms |
| 4 | 1378.0 ms |
| 5 | 1065.4 ms |
| 6 | 1425.6 ms |
| 7 | 1129.3 ms |
| 8 | 1374.7 ms |
| 9 | 1117.5 ms |
| 10 | 1202.4 ms |

- P50：1154.4 ms
- P90：1378.0 ms
- 最大值：1425.6 ms
- 3 秒内完成：10/10
- 超过 5 秒：0/10

P50 阶段明细：

| 阶段 | 累计耗时 |
| --- | ---: |
| 原生首帧 | 192.3 ms |
| WebView 创建请求 | 192.3 ms |
| 核心状态就绪 | 205.2 ms |
| WebView 创建完成 | 849.3 ms |
| 页面导航完成 | 909.8 ms |
| React 挂载 | 971.5 ms |
| 主页面可交互 | 1154.4 ms |
| 后台任务门控释放 | 1154.6 ms |
| watcher 调度 | 1157.4 ms |
| 搜索调度 | 1157.4 ms |
| 托盘就绪 | 1193.6 ms |

结果证明 watcher、搜索、缓存维护和托盘初始化均发生在主页面可交互之后。前端初始资源约 91.55 KB gzip，React 挂载到可交互耗时约 183 ms，不是主要瓶颈，因此本轮不拆分搜索进程或前端模块；若低配置设备的阶段数据表明进程装载或前端执行成为主导，再执行该可选拆分。

## 开发构建补充验证

启动 Vite 后对 `target/debug/logcrate.exe` 连续测量 3 次，结果为 2652.2 ms、1617.4 ms、1500.2 ms，全部低于 3 秒。首个样本包含开发服务器首次模块转换，其余样本稳定在约 1.5–1.6 秒。该结果仅用于确认 `tauri:dev` 不再复现二十秒等待，不替代正式的 Release 验收。
