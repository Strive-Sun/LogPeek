## Context

当前主窗口先由 `softbuffer` 绘制原生进度画面，再从后台线程请求创建一个 1×1 子 WebView。只有 `PageLoadEvent::Finished` 到达后，WebView 才扩展到整个客户区。原生动画最多推进到 90%，但其百分比仅由经过时间模拟，不能反映真实启动阶段。

现有监控和搜索状态已经转入后台初始化，因此约二十秒等待更可能发生在进程装载、WebView2/WebKit 创建、子 WebView 调度或前端页面交接路径。生产前端资源约为 91 KB gzip，不能仅凭 bundle 大小解释全部耗时。实现前必须先取得各阶段的真实数据，再对占主导的阶段做结构性优化。

## Goals / Non-Goals

- Goals:
  - 在受支持的 Windows 与 macOS 常规发布环境中约三秒进入可交互主页面。
  - 原生进度反馈仍在窗口首次出现时立即可见，不重新引入白屏。
  - 首屏不等待监控恢复、搜索、更新或缓存维护。
  - 用本地计时和可重复基准证明优化效果并捕获回退。
- Non-Goals:
  - 伪造更快的进度动画或提前隐藏窗口。
  - 要求三秒内完成所有目录扫描、搜索索引或日志会话恢复。
  - 向远程服务上传启动遥测。
  - 为一次优化引入完整原生 GUI 框架。

## Decisions

### Decision: 以首个可交互主页面作为启动完成点

启动耗时从进程入口开始，到 React 主界面提交后至少完成一个浏览器绘制帧，并向 Rust 报告主导航和监控目录区域已可接收输入为止。`PageLoadEvent::Finished`、React 函数执行或进度条到达 100% 均不能单独代表启动完成。

发布版基准在参考设备上先预热一次运行时，然后连续启动十次：至少九次 MUST 在 3 秒内达到可交互里程碑，任何一次 MUST NOT 超过 5 秒。另行记录系统重启后的冷启动结果，但不把首次安装 WebView 运行时或操作系统更新成本计入应用关键路径。

### Decision: 先测量，再针对主导阶段优化

启动追踪至少包含：`process-start`、`native-first-frame`、`webview-create-requested`、`webview-created`、`navigation-finished`、`react-mounted`、`interactive-frame`、`core-state-ready`、`watchers-scheduled` 和 `search-scheduled`。记录使用单调时钟和本地结构化输出，不进行网络上传，也不在 UI 线程同步写磁盘。

实现首先建立未优化基线。只有测量显示某阶段是主要瓶颈后，才采用对应措施：

- WebView 创建或主线程调度占主导：将 WebView 创建请求提前到原生首帧之后，移出托盘、更新器和业务状态恢复等无关前置步骤，并复核子 WebView 调度方式。
- 前端加载占主导：延迟加载搜索、设置、更新和非首屏对话框，缩小初始执行图，使用前端显式可交互握手完成交接。
- 主进程装载占主导：评估把 Tantivy、SQLite、中文分词和搜索 provider 移入按需搜索进程，使默认关闭搜索时主进程无需装载这些实现。
- 原生动画争用资源：降低无变化帧的绘制频率，避免在 WebView 冷启动期间持续满窗口 60 FPS 软件重绘。

### Decision: 主页面与后台恢复采用分层就绪状态

启动状态分为：

1. `shell-ready`：主界面结构可见且可交互。
2. `core-data-ready`：轻量配置和监控目录列表可用。
3. `background-ready`：watcher、搜索、更新检查和维护任务已调度或完成。

主页面显示只依赖 `shell-ready`。目录列表可使用加载占位并在 `core-data-ready` 后增量填充。搜索和维护任务只能在首屏关键任务已调度后启动，并不得通过 CPU、磁盘或锁竞争显著延迟首屏。

### Decision: 保留真正立即显示的原生启动反馈

继续在 WebView 可用前显示原生进度画面，不通过隐藏主窗口制造“快速启动”。交接由前端 `interactive-frame` 握手驱动；握手前保留原生画面，握手后一次性停止绘制并显示已完成首帧的 WebView，避免白屏和双重加载层长时间叠加。

## Risks / Trade-offs

- 三秒目标受 WebView 运行时、杀毒软件和设备性能影响 → 固定发布构建、参考设备和十次样本规则，同时保留五秒退化上限与冷启动记录。
- 主页面先于目录数据就绪可能出现短暂占位 → 保留稳定布局并增量填充，不能阻止导航和设置等基础操作。
- 将搜索实现移入独立进程改动较大 → 只有进程装载测量证明必要时才执行，并保持现有搜索启停与 IPC 错误语义。
- 前端握手丢失会使原生画面无法交接 → 增加明确错误日志和受控兜底；兜底不能用固定延迟冒充成功。
- 降低动画帧率可能影响观感 → 以平滑但低争用为原则，验证不同 DPI 和双平台表现。

## Migration Plan

1. 加入启动阶段时钟、前端可交互握手和发布版基准，保存当前二十秒路径的阶段数据。
2. 将 WebView 创建与首屏资源调度前移，延后托盘、更新、业务恢复和维护任务。
3. 按测量结果优化前端初始执行图、原生绘制频率或主进程依赖边界。
4. 在 Windows 与 macOS 发布构建各执行十次常规启动及一次冷启动，记录 P50、P90、最大值和阶段明细。
5. 若任一常规样本超过五秒或九成样本未达到三秒，则任务保持未完成并继续定位瓶颈。

## Open Questions

- Windows 参考设备上的二十秒主要发生在进程装载、WebView2 创建还是页面导航阶段，需要阶段计时确认。
- macOS 是否存在相同量级的延迟，需在相同里程碑定义下补充基线。
