## 1. 背景与目标

### 1.1 问题

排查线上或客户端问题时,日志通常以压缩包形式流转(zip / tar.gz / 7z / rar)。当前工作流是:

1. 下载压缩包
2. 手动解压到某个目录
3. 在解压结果里翻找 `.log` / `.txt`
4. 用文本编辑器打开查看

当需要频繁查看大量日志包时,第 2、3 步的手工操作成本很高,且解压产物污染磁盘。

### 1.2 目标

做一个跨 Windows + macOS 的桌面日志阅读器,把上述流程压缩为"下载即看":

- 监控用户指定的目录,自动发现新到达的日志压缩包**与裸文本日志文件**,支持按后缀自由筛选
- **免解压**(不落地到磁盘)流式读取压缩包内的日志文件;裸文本文件直接查看
- 目录树把压缩包"虚构展开"为其内部文件,展开只读清单不解压
- 新日志到达时在顶栏提示,用户一键打开查看
- 能流畅查看 GB 级大日志(行索引 + 窗口化加载,不整包进内存)

### 1.3 非目标(v1 明确不做)

- 手机端(iOS / Android)
- 日志的服务器端聚合 / 上传 / 检索(纯本地工具)
- 日志格式的语义解析(如 log4j 字段结构化)——v1 只做纯文本查看
- 实时远程 tail(SSH / 网络流)

## 3. 技术选型

### 3.1 框架:Tauri(Rust 后端 + Web 前端)

选择理由:

- **大文件能力**:Rust 可精确控制内存,内存映射 / 流式读取 / 零拷贝都很自然,适合 GB 级日志的行索引与窗口化加载。
- **免解压读取**:Rust 压缩库生态覆盖主流格式(见 3.3)。
- **现代交互**:顶栏"新日志提示"这类状态化交互用 Web 前端(React/Vue/Svelte)开发效率最高。
- **包体积 / 内存**:复用系统 WebView,安装包体积和空载内存显著小于 Electron。
- **跨平台**:Tauri 原生支持 Windows + macOS 打包。

代价:需要 Rust,7z/rar 的库生态相对 zip/tar.gz 偏弱(见风险)。

### 3.2 前端

- 框架:**React 18**
- 语言:TypeScript
- UI:Radix UI / shadcn 风格基元 + CSS 变量主题,`@tanstack/react-virtual` 做日志虚拟滚动(详见第 6 章 UI 设计)
- 职责:纯 UI 与交互 —— 目录树、条目列表、顶栏提示、虚拟滚动文本视图。不做任何重 IO / 解压逻辑。

### 3.3 关键 Rust crate

| 用途 | crate | 说明 |
|---|---|---|
| 目录监控 | `notify` | 跨平台封装 FSEvents / ReadDirectoryChangesW |
| zip 读取 | `zip` | 按名定位条目返回解压流;Stored 条目可 seek,Deflate 顺序解压(见 4.4) |
| tar / gzip | `tar` + `flate2` | 流式读取(M3) |
| 7z | `sevenz-rust` | 纯 Rust,可用但相对小众(M3) |
| rar | `unrar`(绑定) | rar 为专利闭源算法,授权受限,v1 可延后(M4) |
| 通用归档抽象 | 自封装 trait | 屏蔽格式差异,见 4.3 |
| 文本编码 | `encoding_rs`(+ 探测启发式) | GBK/GB18030/UTF-16/UTF-8 解码与 BOM 识别(见 4.4) |
| 配置持久化 | `serde` + `serde_json` | 监控目录列表等本地配置序列化 |
| 内存映射 | `memmap2` | 大文件行索引 |
| 异步运行时 | `tokio` | 后台索引 / 监控任务 |

### 3.4 安装包体积预期

选 Tauri 而非 Electron 的核心理由之一即包体积。Tauri 复用**操作系统自带 WebView**(Windows→WebView2,macOS→WKWebView),不打包 Chromium/Node.js 运行时,因此安装包比同功能 Electron 应用小一个数量级。

#### 预期体积

| 平台 | 安装包体积 | 说明 |
|---|---|---|
| Windows | ~6–10MB | 前提系统已有 WebView2(Win10 1803+ / Win11 基本预装);离线内嵌 WebView2 则 +约 150MB |
| macOS | ~10–15MB | WKWebView 为系统组件无需附带,.dmg 打包 |

体积构成(在 Tauri 极简基线 Win ~2.5MB / mac ~4MB 之上的功能增量):

- 压缩库(`zip`,后续 `flate2`/`tar`)静态链接:+1–3MB
- 文件监控 `notify` + 异步运行时 `tokio`:+1–2MB
- 前端资源(React + 虚拟滚动 + UI)压缩后:+1–3MB
- 行索引 / 内存映射:体积可忽略

横向对比(同等功能):Tauri ~6–15MB / Qt+C++ ~15–40MB / Electron ~85–150MB。

> 注:体积小不代表内存小。WebView 本身仍占内存,空载预计数十 MB;打开 500MB 日志时内存主要花在行索引(约 40MB)而非文件内容(窗口化只加载可视区)。体积与内存是两个独立指标。

### 3.5 Windows WebView2 打包策略

WebView2 在 Windows 上有三种打包方式,直接影响安装包体积与离线可用性:

| 策略 | 体积影响 | 适用场景 |
|---|---|---|
| **在线引导(默认,推荐)** | 最小(~6MB) | 用户机器缺 WebView2 时安装期联网下载。Win10 1803+/Win11 基本预装,绝大多数用户无感 |
| 离线内嵌 | +约 150MB | 面向内网 / 无外网环境,完全离线可装 |
| 固定版本 | +约 180MB | 绑定特定 WebView2 版本,用于对 WebView 版本极端敏感的场景 |

**决策**:LogCrate 面向开发者工具、目标用户机器普遍为较新 Win10/11,采用**默认在线引导**,使安装包保持在 10MB 以内。macOS 无此选择(WKWebView 恒为系统组件)。

### 4.2 目录监控(Watcher)

- 用 `notify` 监听用户配置的一个或多个目录。
- 收到"新文件"事件后,先做**稳定性检测**:文件可能还在下载/写入中,需轮询其大小,在连续稳定窗口内(如 500ms×3)大小不再变化后判定"到达完成"。不采用平台 close 事件方案——`notify` 跨平台封装不完整、行为不一致,大小稳定检测更可移植(已在提案 design.md 定案)。
- 完成后,判定文件类型。M1 受支持两类:**zip 归档**(按扩展名/magic number)与**裸文本日志文件**(按扩展名或内容采样为文本)。裸文本无需解压,直接可查看。
- 通过**可配置的后缀筛选规则**(如仅 `.log`/`.txt`/`.out`)过滤:不匹配的文件不进目录树、不计入通知。筛选规则与监控目录列表一同持久化。
- 判定为受支持且通过筛选 → 通过 Tauri event 向前端推送"有新日志"通知,前端在顶栏显示提示。

关键点:**不自动解压、不自动打开**。只提示。用户点开时才触发 4.3 的读取。这样避免大包到达即卡顿。

**目录树惰性展开**:高频场景下监控目录可能有几百个包。目录树把每个 zip 呈现为可展开节点,但**只在用户展开某个包时才**读其中央目录列条目(见 4.3),绝不在扫描阶段预先打开所有包。裸文本文件则是可直接点击的叶子节点。

### 4.3 归档读取(Archive)—— 免解压核心

定义统一抽象,屏蔽格式差异:

```rust
/// 归档内的一个条目(不含内容,仅元信息)
struct ArchiveEntry {
    path: String,       // 包内路径
    size: u64,          // 解压后大小(中央目录声明值,仅供展示,不作为安全上限依据,见 4.4)
    is_log: bool,       // 是否是日志/文本(按扩展名+采样判定)
    encrypted: bool,    // 是否加密条目(M1 不支持,列出但标记)
}

/// 归档读取器抽象。不同格式实现同一接口。
trait ArchiveReader {
    /// 列出包内条目,不解压内容
    fn entries(&self) -> Result<Vec<ArchiveEntry>>;
    /// 打开某个条目,返回一个可流式读取的 Read 流
    fn open_entry(&self, path: &str) -> Result<Box<dyn Read + Send>>;
}
```

- **zip**:`zip` crate 支持按名定位条目并返回解压 `Read` 流(Stored 条目零拷贝定位,Deflate 条目顺序解压)。注意随机跳转能力取决于条目压缩方式,详见 4.4。
- **裸文本文件(passthrough,M1)**:监控目录里的 `.log`/`.txt` 等裸文件用一个直通实现,`entries()` 返回单个条目(文件自身),`open_entry()` 直接返回文件句柄。这样目录里的裸文件与包内条目走**完全相同**的查看路径。裸文件天生完全可 seek,不需要解压、不需要临时缓存,是最简单的一种。
- **tar.gz**:顺序流,`open_entry` 需从头流式扫描到目标条目;`entries` 需要扫一遍(可缓存条目偏移)。(M3)
- **7z / rar**:各自库实现同一 trait。rar 视授权情况 v1 可先不做。(M3/M4)

**列条目 ≠ 解压(关键)**:`entries()` 与 `open_entry()` 是两件独立的事。zip 末尾有一段**中央目录**清单,记录每个条目的名字、路径、解压后大小、压缩方式、包内偏移——这些元信息未被压缩,可直接读。因此列出条目(即目录树"虚构展开",见 4.2/6.3)**只 seek 读末尾几 KB~几十 KB 的清单,不解压任何内容、不产生临时文件、不占磁盘**。一个 500MB 的 zip,展开其目录树可能只读了末尾 20KB。解压与可能的临时缓存**只在用户真正点开某个具体条目查看时**发生,且只针对那一个条目。

**免解压的含义**:内容不落地到磁盘,而是以 `Read` 流的形式交给行索引层按需消费。用户看哪个文件才解哪个,且是流式解、不整体展开。

**安全边界**:归档由用户从外部下载,内容不可信,读取时施加以下防护:

- **大小上限以实际读取字节数为准**:不信任中央目录声明的解压后大小(可被伪造以绕过上限,即 zip bomb)。建索引流式读取时累计实际字节,超过上限(M1 为解压后 2GB)立即中止并返回错误。
- **不支持的特性明确报错**:加密条目、分卷归档在 M1 不支持,列出时标记并在打开时返回明确错误,不崩溃、不悬挂。
- **路径安全**:列条目时对包内路径做规范化,防止 `../` 穿越(即便 M1 不向磁盘写出解压产物,内部临时缓存的落盘路径也须限定在受控临时目录内)。

### 4.4 行索引与窗口化加载(LineIndex)—— 大文件核心

GB 级日志不能整包进内存。方案:

1. **建索引阶段**:后台流式读取条目内容,记录每一行(或每 N 行一个采样点)的字节偏移,存成一个偏移数组。索引本身远小于原文件(1GB / 每行 100 字节 ≈ 1000 万行,偏移用 u64 存约 80MB;2GB 上限约 160MB;采样点 + 段内扫描可进一步压缩,留待 M2)。
2. **查看阶段**:前端虚拟滚动只请求可视区的行范围(如第 10000~10050 行),后端按行偏移读取这几十行返回。

**关键澄清:zip 条目的"随机访问"分两种情况**,不能一概而论:

- **Stored(未压缩)条目**:解压后字节 = 归档内字节,可对归档文件直接 `seek` 到目标偏移读取,真正零落地随机访问。
- **Deflate(压缩)条目**:这是 zip 的默认压缩方式。**解压流本身是顺序的**——无法在解压中途 seek 到第 N 字节,任意跳转都要从条目头重新解压。因此对压缩条目,"记录解压后偏移 + 直接 seek"这一朴素方案在物理上不成立。

**M1 对压缩条目的落地策略(方案 A)**:建索引阶段一次性顺序解压该条目,把解压结果写入**内部临时缓存文件**(非用户可见的"解压到文件夹"),同时记录行偏移;查看阶段对该缓存文件 seek 读取。缓存在 session 关闭时清理。这与 tar.gz 等顺序流格式的处理是同一套机制。

> **磁盘代价与取舍(几百 MB 场景重点)**:临时缓存占用等同**解压后大小**。目标场景的压缩包可达几百 MB,按 5~10 倍缩比解压后可能 1GB+;多个会话叠加(LRU 缓存)磁盘占用可达数 GB。通过 LRU 会话上限 + 关闭即清 + 进程退出兜底控制;写盘失败则回退为仅顺序读并提示。这是"打开后随机访问无感"与"零磁盘占用"之间的取舍,M1 选前者。
>
> **M2 优化方向——Deflate 重启点索引**:对 Deflate 流每隔若干 MB 解压输出保存一个 inflate 器状态快照(约 32KB window),随机跳转时从目标前最近的重启点恢复、只解一小段(≤ 段间距)即到位。**免磁盘放大**(不落盘全量解压结果),内存仅增若干 MB 重启点。参考 zlib 官方 `zran.c`。因需底层 inflate 状态存取、复杂度高,M1 先用落盘缓存快速跑通,此项排入 M2。

#### 边建边读

索引在后台顺序建立,已建立部分即刻可读:前端请求的行范围若超出当前已索引行数,后端返回已索引部分的可用行并附带当前已索引行数上界(`indexed_lines`),不阻塞等待整体完成。这样用户打开大文件可立即查看开头,无需等全量索引。

#### 解压/跳转的"无感"体验目标

产品目标是让打开、随机跳转、滚动阅读对用户尽量无感。实现思路是**"一趟解压把压缩流摊平成可随机访问的缓存"**,而非在压缩流里跳(Deflate 物理上做不到):

- **一次性解压到缓存**:打开 Deflate 压缩条目时,后台跑一趟顺序解压,边解边写入内部临时缓存文件、边记录行偏移。**这趟跑完后,该条目等价于一个普通未压缩文件**,随机跳转/滚动都是对缓存文件 seek,与裸文本文件同速,彻底无感。裸文本文件与 Stored 条目无需这趟,天生可 seek。
- **滚动阅读必然无感**:解压速度约 100~200MB/s,远快于人眼滚动阅读;解压前沿始终领先阅读位置,顺序阅读全程无阻塞。这是最主要的使用路径。
- **随机跳转基本无感**:中小文件(几 MB~几十 MB)整趟解压 sub-second,打开瞬间全文件即可随机跳;大文件(解压后数百 MB~2GB)整趟解压需数秒~十几秒,跳到已解压区即时,跳到未解压远端需等前沿推进。
- **唯一有感边界**:打开一个大的 Deflate 条目后,在后台解压完成前立即跳到文件最末尾——需从头解压到末尾,有数秒 + 进度反馈(而非假死)。这是 Deflate 流不可随机 seek 的物理限制,任何工具皆然(lnav/klogg 对压缩流同理)。M2 的重启点索引可把这一边界从"从头解压"缩短为"从最近重启点解一小段"。

体验指标(待验证项中定值/实测):中小文件点开到首屏可见 < ~200ms;顺序滚动无掉帧;已就绪范围随机跳转无可感知延迟。

#### 文本编码与行分隔符

- **编码**:日志常见 GBK/GB18030(Windows 中文)、UTF-16(含 BOM)、UTF-8(含/不含 BOM)。后端在建索引前先探测编码(`encoding_rs` 解码 + 轻量启发式/BOM 识别),`read_lines` 返回时统一解码为 UTF-8 文本交前端。检测不准时前端可手动指定编码重新解码刷新。
- **行分隔符**:兼容 LF(`\n`)与 CRLF(`\r\n`)建立行偏移索引;返回行内容时剥离行尾 `\r`/`\n`。注意行偏移按**字节**记录,与编码/换行解码解耦(索引记原始字节位置,读取时再解码)。

#### 超长行处理

单行可能有数 MB(如整行 JSON / stacktrace)。后端 `read_lines` 对超过阈值(如 64KB)的行截断并标记,避免一次性传输超大单行冲击内存与 IPC;前端对截断行提供横向虚拟滚动或"展开完整行"操作。行读取返回结构:

```rust
/// 一行返回给前端的内容
struct LogLine {
    line_no: u64,       // 行号(从 0 或 1 起,前后端约定一致)
    content: String,    // 已解码为 UTF-8 的行内容(可能被截断)
    truncated: bool,    // 是否因超过阈值被后端截断
}
```

### 4.5 查看会话生命周期(Session)

`open_log_session` 建立一个查看会话,持有该条目的行偏移索引(约 80MB/GB)以及 Deflate/顺序流条目的内部临时缓存文件(约等于解压后大小)。几百 MB 压缩条目的缓存可达 1GB+,故会话资源管理在此场景尤为关键:

- **显式释放**:前端关闭或切换离开视图时调用 `close_log_session`,释放索引并删除该会话的临时缓存。
- **LRU 回收**:同时保留的会话数设上限(如 3~5 个),超限时释放最久未使用的会话;被释放的会话再次访问需重新建索引。
- **进程退出兜底**:应用退出时清理所有残留临时缓存,防止磁盘泄漏。

> 会话上限与临时缓存目录为待定参数,实现前定值(见第 7 章待验证项)。

### 4.6 前后端接口

**命令(前端 → 后端,invoke)**:

| 命令 | 参数 | 返回 |
|---|---|---|
| `add_watch_dir` | `path` | ok |
| `remove_watch_dir` | `path` | ok |
| `list_watch_dirs` | - | `Vec<String>`(持久化的监控目录) |
| `list_archive_entries` | `archive_path` | `Vec<ArchiveEntry>` |
| `open_log_session` | `archive_path, entry_path` | `session_id`(建索引后返回) |
| `read_lines` | `session_id, start, count` | `Vec<LogLine>`(含截断标记) |
| `close_log_session` | `session_id` | ok(释放索引与临时缓存) |
| `search`(M4) | `session_id, pattern` | 命中行号列表 |

> 命名说明:命令层用 `open_log_session` / `close_log_session`,与归档层 `ArchiveReader::open_entry`(返回 `Read` 流)区分——前者建索引并管理查看会话,后者只提供解压流。`search` 为 M4 能力,此处仅预留占位,M1 不实现。`read_lines` 返回的每行携带是否被后端截断的标记(见 4.4 超长行处理)。

**事件(后端 → 前端,emit)**:

| 事件 | 载荷 | 用途 |
|---|---|---|
| `new-archive-detected` | `{path, size}` | 顶栏提示新日志(不含条目数,见下方说明) |
| `archive-removed` | `{path}` | 监控目录中日志包被删除 → 前端从通知列表移除并更新计数 |
| `archive-updated` | `{path, size}` | 同名日志包被覆盖且通过到达检测 → 前端重新计入未查看 |
| `index-progress` | `{session_id, percent, indexed_lines}` | 大文件建索引进度条;`indexed_lines` 为当前已索引行数,支撑边建边读 |

> `new-archive-detected` 载荷刻意**不含** `entry_count`:到达检测阶段遵循 4.2"只提示不读取"原则,不打开 zip 读中央目录;条目数与列表在用户点开时通过 `list_archive_entries` 惰性获取。

## 5. 关键流程

### 5.1 新日志包到达

```
用户下载 log.zip 到监控目录
  → notify 触发 create 事件
  → Watcher 等待文件大小稳定(下载完成)
  → 判定为受支持归档格式
  → emit "new-archive-detected"
  → 前端顶栏出现红点提示 "1 个新日志包"
  → 用户点击 → 展开包内文件列表(list_archive_entries,免解压)
```

### 5.2 打开并查看大日志

```
用户点包内某个 .log
  → invoke open_log_session
  → 后端流式读取该条目,后台建行偏移索引(emit index-progress,含已索引行数)
  → 返回 session_id(边建边可读)
  → 前端虚拟滚动,按需 invoke read_lines(session_id, 可视行范围)
  → 后端按偏移读取对应行(解码为文本)返回
  → 用户关闭/切换 → invoke close_log_session 释放索引与临时缓存
```

## 7. 风险与待验证项

| 风险 | 说明 | 应对 |
|---|---|---|
| 7z/rar 库成熟度 | Rust 生态弱于 zip/tar | v1 优先 zip + tar.gz;7z 用 `sevenz-rust` 试点;rar 视授权延后 |
| Deflate 流无法随机 seek | zip 压缩条目 / tar.gz 内大文件跳转难 | 透明内部临时缓存(4.4) |
| 文件到达完整性判断 | 下载中文件误判为已到达 | 大小稳定检测(轮询大小,连续稳定窗口内不变) |
| 前端渲染超长行 | 单行超长导致卡顿 | 后端截断标记 + 前端横向虚拟滚动(4.4) |
| 索引内存占用 | 超大文件偏移数组大 | 采样点 + 段内扫描降低索引密度 |
| 恶意/异常归档 | zip bomb、加密/分卷、路径穿越 | 按实际字节熔断上限、明确报错、路径规范化(4.3) |
| 日志文本编码多样 | GBK/UTF-16 日志乱码 | `encoding_rs` 探测解码 + 手动指定编码(4.4) |

### 待验证项(实现前需定值/验证)

- **参数定值**:大小稳定检测窗口(暂定 500ms×3)、大文件上限(暂定解压后 2GB)、超长行截断阈值(暂定 64KB)、并发会话上限(暂定 3~5,直接影响临时缓存磁盘峰值)、内部临时缓存目录位置与总容量上限。
- **无感体验指标实测**:中小文件点开到首屏可见 < ~200ms、顺序滚动无掉帧、已就绪范围随机跳转无可感知延迟——需在真实机器与典型日志上实测确认,并据此校准解压/索引的后台调度。
- **索引/解压缓存持久化**:**结论——不做**。目标场景是"高频下载新文件"而非"高频重复打开同一文件",重复打开率低,持久化收益小却引入缓存失效、磁盘清理等复杂度。且真正耗时在解压而非建索引,叠加"边建边读"后首屏已不阻塞。仅当未来实测出现"反复打开同一巨型包、每次等解压"的高频反馈,才重新评估(届时应缓存解压结果+索引,而非只存索引)。
- **编码探测准确率**:启发式探测对短文件/混合编码可能误判,需实测并确保手动切换可兜底。
- **路径安全 / 命令权限**:`archive_path` 由前端传入,需结合 Tauri capability 限制可访问路径范围,防止越权读取任意文件。

## 8. 里程碑(MVP 优先级）

**M1 — 最小闭环(zip + 裸文本,含大文件查看)**
- 多目录监控(持久化,重启恢复)+ 可配置后缀筛选 + 新文件顶栏提示(计数逐条递减 + 一键清零)
- zip 免解压列条目 + 打开条目;裸文本文件 passthrough;目录树惰性展开
- 行偏移索引 + 虚拟滚动窗口化加载,支持解压后 ≤ 2GB 流畅查看(方案 A:压缩条目落盘缓存)
- 建索引进度提示 + 边建边读 + 打开/跳转/滚动无感体验
- 深浅色主题骨架(CSS 变量 + 跟随系统/手动切换,配色打磨见 M4)
- 文本编码检测与解码(UTF-8 / GBK / UTF-16 + BOM)

**M2 — 超大文件与体验**
- **Deflate 重启点索引**(免磁盘放大的随机访问,替代方案 A 的落盘缓存;参考 zlib `zran.c`)
- 采样索引优化(降低索引密度,支撑更大文件)
- "已看"状态持久化

**M3 — 多格式**
- tar.gz 支持(含临时缓存策略)
- 7z 试点

**M4 — 体验打磨**
- 搜索/过滤、高亮、主题配色打磨
- rar(视授权)
